## Core Project Principles

### Notebook-first, always
This repository is **notebook-first at all times** unless Ronnie explicitly decides otherwise.

That means:

- notebooks are the primary surface for experimentation
- notebooks are the default place for model development
- notebooks should support rapid debugging and quick iteration
- notebook workflows should remain understandable and inspectable
- abstractions should serve the notebook workflow, not replace it

Do not force premature migration into heavyweight packages, frameworks, CLIs, or rigid abstractions if doing so slows iteration or makes debugging harder.

### Research clarity over software theater
Favor:

- transparent logic
- explicit variables
- clear plots
- direct diagnostics
- readable experiments
- interpretable outputs

Avoid unnecessary complexity introduced only to make the repo look more “production-grade.”

### Preserve semantics
Named physical fields, explicit metrics, structured reports, and architecture-specific diagnostics are important. Do not collapse meaningful scientific or modeling structure into anonymous pipelines without a strong reason.

### Build for understanding
The repo is not only for producing outputs. It is for learning how systems work.

Prefer work that improves understanding of:

- why a model succeeds or fails
- what a diagnostic means
- how training behaves
- what constraints matter
- how architecture choices change outcomes

### Keep the work cumulative
Agents should prefer extensions that naturally build on what already exists rather than introducing disconnected side projects.

## How to Reason About Tradeoffs

When making tradeoffs, agents should generally prioritize in this order:

1. preserves notebook-first iteration and debugging
2. improves technical understanding
3. improves scientific or systems rigor
4. strengthens portfolio and application value
5. improves extensibility without overengineering
6. improves aesthetics and cleanup

A cleaner abstraction is not better if it slows learning.
A more complex system is not better if it obscures understanding.
A larger model is not better if it adds cost without new insight.
